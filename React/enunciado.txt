> Tema: Batalha Naval Avançada
O tema do Trabalho Prático é uma variante do jogo clássico “Batalha Naval”, com uma vertente
tática e de gestão de recursos críticos. O jogo decorre entre dois adversários (o Jogador e o
Computador), onde cada um possui uma frota de navios distribuída num tabuleiro. O objetivo é
destruir toda a frota adversária, gerindo os recursos disponíveis.
A frota será distribuída num tabuleiro quadrado de dimensão mínima 10x10. Após o
posicionamento dos navios, o jogo inicia, com os jogadores a atacarem alternadamente uma
célula do adversário até que toda a frota de um deles seja destruída.
Nesta versão de Batalha Naval, existem características especiais:
→

Gestão de Combustível - O jogador inicia com 100 unidades de combustível (valor máximo
permitido). Cada disparo consome 5 unidades. Se acertar num navio inimigo, recupera 10
unidades (até ao limite de 100). Se o combustível chegar a 0, o jogador perde imediatamente
o jogo.

→

Cronómetro de turno - O jogador tem um limite de 15 segundos para realizar cada jogada.
O cronómetro deve ser visível, atualizar em tempo real em contagem decrescente (15 a 0).
Se o tempo esgotar, o turno passa automaticamente para o computador e o jogador é
penalizado em 5 unidades de combustível.

→

Radar Inteligente (recompensa) - A opção de "Radar" fica disponível apenas sob condições
de mérito: o jogador deve acertar num navio inimigo numa jogada realizada em menos de 3
segundos. Ao ativar o radar (ação livre que não consome o turno nem combustível), o
sistema destaca no tabuleiro do computador uma área de 2x2 que contenha, pelo menos,
uma parte de um navio ainda não atingida. O radar é de uso único por cada ativação obtida.

> Funcionalidades Obrigatórias
O jogo deverá ser implementado em React 19, utilizando as melhores práticas (Componentes
Funcionais, Hooks) e disponibilizar, pelo menos, as seguintes funcionalidades:
→

Configuração Inicial: Solicitar o nome do jogador antes do início da partida.

→

Posicionamento da Frota: Apresentar o tabuleiro vazio e permitir o posicionamento manual
da frota do jogador por interação de clique. Os navios podem ser posicionados horizontal ou
verticalmente, não sendo permitida a sobreposição nem o posicionamento fora dos limites
do tabuleiro. A frota obrigatória é composta por:
-

1 navio de tamanho 5
1 navio de tamanho 4
2 navios de tamanho 3
2 navios de tamanho 2

→

Configuração do Computador: Deverá existir uma lista/menu que permita ao utilizador
selecionar entre 3 tabuleiros com frota pré-definida para o computador, ou escolher uma
opção de "frota aleatória" (onde o sistema gera um posicionamento válido de forma
autónoma).

→

Modo Debug/Testes:
Deverá existir uma caixa de seleção (checkbox) para apresentar/esconder a frota invisível do computador. Isto servirá para facilitar a correção,
testes rápidos e verificação do funcionamento correto do jogo por parte do docente.
→

Interface de Jogo: O sistema deverá distinguir e apresentar visualmente as seguintes fases
das células ao longo do jogo:

Água (Não explorada)

-

Tiro Falhado (Água explorada)

-

Acerto (Parte do navio atingida)

-

Navio Destruído (Totalmente afundado)

Painel de Informações:
-

Apresentar o Cronómetro de Turno (15s decrescentes).

-

Apresentar o Nível de Combustível atual, atualizando-o de acordo com os disparos e
penalizações.

-

Mostrar o botão do Radar como "Disponível" ou "Indisponível", dependendo da
satisfação da regra de mérito.

→

Inteligência do Computador: Quando é o computador a atacar, a implementação da sua
lógica de ataque terá dois níveis de valorização:
-

Nível Básico (Menos valorizado): O computador ataca sempre numa posição
puramente aleatória (desde que não tenha sido previamente atacada).

-

Nível Avançado (Mais valorizado): O computador ataca aleatoriamente até
encontrar um navio. Assim que acerta numa célula com navio, os ataques seguintes
deverão ser feitos nas posições adjacentes (horizontal e vertical) até que esse navio
seja totalmente destruído. Só depois de o afundar é que retoma a procura aleatória.

→

Fim de Jogo: Quando houver um vencedor (ou o combustível acabar), deverá ser
apresentado um ecrã final com:
-

O Vencedor.

-

O número de jogadas efetuadas.

-

Um botão para "Jogar Novamente", que reinicia os estados todos para uma nova
partida.

> Implementação
O trabalho deverá ser implementado em React 19, recorrendo exclusivamente a Componentes
funcionais e Hooks React. É dada a liberdade ao aluno para seguir a estratégia que considerar
melhor, no entanto, especificam-se abaixo as características e limitações obrigatórias:
→

Base de Projeto: Os alunos devem usar obrigatoriamente base do projeto fornecido nas
aulas práticas;

→

Arquitetura de Componentes: A aplicação não pode ser um ficheiro monolítico. É
obrigatória a separação lógica da interface em múltiplos componentes com
responsabilidades bem definidas. Exige-se, no mínimo, a divisão da interface nos seguintes
componentes (ou equivalentes):
−

Componente para o Tabuleiro (Board)

−

Componente para a Célula individual do tabuleiro (Cell / Square)

−

Componente para o Painel de Informações (Dashboard / ControlPanel - para gerir
cronómetro, combustível e radar)

→

−

Componente para a fase de Posicionamento da Frota (Setup)

−

Componente para o Ecrã Final (GameOver / Results)

A gestão de estados (useState) não deve ser centralizada inteiramente no componente
principal (App).

→

Não é permitido o uso de useReducer, Context API ou bibliotecas externas (Redux, etc).

→

Não é permitida manipulação direta do DOM , sob pena de anulação da cotação do respetivo
requisito. São expressamente proibidos:
−
−
−
